# 标准化考试系统 PRD

## 1. 背景与目标

### 1.1 项目背景

当前项目定位为一个标准化考试系统，兼容雅思机考的考试形式，用于承载多科目在线考试流程与统一的考试交互体验。

项目现阶段已经形成较完整的前端原型，覆盖听力、阅读、写作、口语四科考试页面，以及考前确认、考试完成页等流程页面。本 PRD 基于当前原型和已确认范围进行收敛，作为后续正式产品化与系统化实现的需求基线。

本项目同时继承现有平台的标准化考试生命周期，在该生命周期基础上增加雅思配套能力，而不是重新设计一套完全独立的考试状态体系。<mccoremem id="03g0s7f0ojb0ewv88wk3gy3bl" />

### 1.2 项目目标

| 目标项 | 说明 |
| --- | --- |
| 系统定位 | 构建标准化考试系统，兼容雅思机考形式 |
| 体验目标 | 提供统一、稳定、接近真实机考的考试流程和交互体验 |
| 规则目标 | 听力、阅读、写作、口语四科的题量、倒计时、提交规则严格对齐官方考试 |
| 数据目标 | 实现答题记录、考试进度、完成状态的跨页面保持与后端存储 |
| 能力目标 | 支持口语模块实时对接信息 |
| 集成目标 | 在现有标准考试状态流上增加雅思配套配置、执行、判分和报告能力 |

### 1.3 本期结论

- 本项目不是“考试系统原型集 + IELTS 主题子系统”，而是一个正式的标准化考试系统。
- 学科考试演示页不纳入当前正式产品范围。
- 监控能力已由现有系统实现，本 PRD 不讨论其设计与实现。
- 听力、阅读、写作的顺序不固定写死，复用现有组合卷能力，由考试配置决定执行顺序。<mccoremem id="03g0s7f0ojb0ewv88wk3gy3bl" />
- 口语优先按“同一考试全部考完”的目标设计，在业务上与主考试记录保持统一归属。<mccoremem id="03g0s7f0ojb0ewv88wk3gy3bl" />

## 4. 业务流程

### 4.1 主流程

1. 用户进入考前确认页。
2. 用户展开目标科目，查看时长和考试说明。
3. 用户点击进入考试，系统进入对应科目的考试页面。
4. 系统按考试配置确定当前组合卷的科目执行顺序。<mccoremem id="03g0s7f0ojb0ewv88wk3gy3bl" />
5. 用户按官方规则完成各科考试流程。
6. 若口语被纳入同一考试实例，则用户在同一考试中完成全部科目后结束。<mccoremem id="03g0s7f0ojb0ewv88wk3gy3bl" />
7. 系统在考试过程中持续保存答题记录、考试进度和完成状态。
8. 用户完成提交后，系统更新该科完成状态。

### 4.2 标准状态流与雅思配套

平台沿用统一的标准考试状态流：

`Draft -> Published -> NotStarted -> InProgress -> Submitted / AutoSubmitted -> Marking -> Scored -> Completed -> Archived`

在该状态流基础上，雅思考试增加如下配套规则：

| 状态阶段 | 雅思配套要求 |
| --- | --- |
| Draft | 增加雅思考试类型、四科结构、Part 规则、时长、提交规则校验 |
| Published | 发布前需校验官方题量、倒计时、Part 结构和组合卷顺序配置 |
| InProgress | 按考试配置的科目顺序依次执行，可包含口语实时考试 |
| Submitted / AutoSubmitted | 支持单科提交与整场考试提交状态归集 |
| Marking | 采用混合判分，听力/阅读自动判分，写作/口语进入人工或 AI 辅助阅卷 |
| Scored | 生成单科结果、总成绩及报告展示数据 |
| Completed | 展示考试完成状态与雅思成绩报告 |
5. 系统在考试过程中持续保存答题记录、考试进度和完成状态。
### 4.3 听力流程
1. 用户从听力 `Part 1` 进入考试。
2. 用户可按规则进入后续 `Part`。
3. 每次切换到新 `Part` 后，系统按官方规则展示读题倒计时。
4. 倒计时阶段允许看题，不允许正式作答。
5. 倒计时结束后恢复作答。
6. 用户完成全部内容后提交，系统记录完成状态并进入完成页。

### 4.3 阅读流程
### 4.4 阅读流程
1. 用户进入阅读 `Part 1`。
2. 用户阅读文章并完成题目作答。
3. 用户在各 `Part` 之间切换，系统保持答题记录。
4. 用户完成所有阅读内容后提交。

### 4.4 写作流程
### 4.5 写作流程
1. 用户进入写作 `Task 1`。
2. 用户查看题干并在编辑器中作答。
3. 用户切换至 `Task 2` 继续作答。
4. 系统记录作答内容与字数状态。
5. 用户完成后提交。

### 4.5 口语流程
### 4.6 口语流程
1. 用户进入口语考试。
2. 系统按考试规则推进题目与 Part。
3. 口语模块实时对接信息，并向用户反馈实时状态。
3. 口语模块实时对接信息，并向用户反馈实时状态。
4. 若口语被编排在组合卷内，则在同一考试记录下继续推进流程并汇总结果。<mccoremem id="03g0s7f0ojb0ewv88wk3gy3bl" />
5. 用户完成口语作答后进入下一 Part 或结束页。

## 5. 功能需求

### 5.1 通用考试框架

| 功能 | 需求描述 |
| --- | --- |
| 统一头部 | 所有考试页使用统一头部样式、账号信息和退出入口 |
| 统一右侧区域 | 展示考试状态、剩余时间、监控区、答题卡/工具箱 |
| 统一主题 | 图标颜色、布局、导航和答题卡风格统一 |
| 页面切换 | 支持科目内 Part 切换与考试流程跳转 |
| 标准状态流复用 | 所有雅思考试实例复用平台统一生命周期状态 |

### 5.2 考前确认

| 功能 | 需求描述 |
| --- | --- |
| 科目卡片 | 展示 Listening、Reading、Writing、Speaking 四科 |

| 时长展示 | 展示对应考试时长 |
| 进入考试 | 用户可从卡片进入对应科目考试 |
| 顺序感知 | 展示或承接当前组合卷配置的考试顺序 |

### 5.3 听力模块

| 功能 | 需求描述 |
| --- | --- |
| Part 结构 | 按官方规则组织听力 Part 数量、题量和流程 |

| 分组展示 | 题目选择区支持按 Part 分组滚动展示 |
| 提交检查 | 提交前检查未答题数量并提示 |
| 完成回流 | 提交后进入完成状态页，并更新考试状态 |
| 顺序编排 | 听力在整场考试中的位置由组合卷配置决定 |

### 5.4 阅读模块

| 功能 | 需求描述 |

| Part 切换 | 支持阅读各 Part 的切换 |
| 辅助工具 | 支持前端标注、高亮、段落标签和工具箱笔记 |
| 顺序编排 | 阅读在整场考试中的位置由组合卷配置决定 |

| 顺序编排 | 写作在整场考试中的位置由组合卷配置决定 |

### 5.6 口语模块

| 功能 | 需求描述 |
| --- | --- |
| 实时对接 | 口语考试需实时对接信息 |

| 功能 | 需求描述 |
| --- | --- |

### 7.1 核心数据对象

| 科目顺序配置 | 是 | 记录组合卷中各科执行顺序 |

### 7.2 数据流要求

| 数据流 | 要求 |
| --- | --- |
| 页面跳转 | 页面切换后可恢复考试进度与已答记录 |
| 提交更新 | 提交成功后立即更新完成状态 |
| 状态回显 | 考前确认页、完成页可读取后端状态 |
| 口语实时数据 | 需支持实时接收、发送与状态反馈 |
| 科目编排 | 需按组合卷配置恢复和推进当前考试顺序 |
| 成绩报告 | 需支持按四科维度生成报告展示数据 |

- 后端答题记录的数据模型待确认。

- Speaking 与主考试记录的关联键设计待确认。
- 缺考场景下的口语状态与汇总规则待确认。

## 8. 权限与状态

### 8.1 角色范围

当前 PRD 默认以考生角色为核心，不扩展后台管理角色设计。

### 8.2 状态定义

| 状态类型 | 状态值 |
| --- | --- |
| 科目状态 | 未开始、进行中、已完成 |
| 作答状态 | 未答、已答、当前题 |
| 提交状态 | 未提交、已提交 |
| 恢复状态 | 可恢复、不可恢复 |
| 口语状态 | 待开始、实时交互中、已结束 |
| 阅卷状态 | 待自动判分、待主观阅卷、评分完成 |

## 9. 异常处理

| 异常场景 | 处理要求 |
| --- | --- |
| 页面刷新或切换 | 可恢复答题记录与考试进度 |
| 提交中断 | 应保留当前答题状态并支持重试 |
| 状态丢失 | 应以后端记录为准恢复状态 |
| 口语实时链路异常 | 应提示用户并提供可恢复或重试机制 |
| 标注/批注丢失 | 不作为考试核心数据处理 |
| 缺考 | 应明确缺考科目状态、是否允许整场完成及结果展示规则 |

以下异常处理细则当前仍需补充为正式方案：

- 自动提交时机与策略 `待确认`
- 网络异常下的缓存与补提交流程 `待确认`
- 口语实时链路失败后的降级策略 `待确认`
